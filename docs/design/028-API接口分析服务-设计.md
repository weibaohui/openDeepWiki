# 1. 总体方案
新增 API 接口分析 Service，从仓库范围 TaskHint 中筛选 API 相关证据，构建 YAML 线索输入，使用“探索 Agent + document_checker + markdown_checker”顺序执行生成 Markdown 文档，并通过 DocumentService 保存为版本化文档。

# 2. 数据与输入结构

## 2.1 证据筛选规则
- 命中关键词：api/route/router/controller/handler/http/接口/路由
- 关键词匹配范围：Aspect、Source、Detail、Title
- 证据检索范围：按 repository_id 在仓库范围内模糊搜索
- 未命中关键词时不纳入 YAML

## 2.2 YAML 输入结构
```yaml
hints:
  - title: "接口线索"
    detail: "router.go 注册了 /api/v1/users"
    source: "backend/internal/router/router.go"
```

# 3. 服务端流程
1. 前端 RepoDetail 页面提供“API接口分析”按钮
2. 接口触发后创建任务并异步执行
3. 按 repoID 在仓库范围检索 TaskHint 并进行关键词筛选
4. 构建 YAML 线索字符串
5. 设置 ADK Session 值（local_path、document_title、task_id）
6. 通过 BuildSequentialAgent 构建“探索 Agent + document_checker + markdown_checker”
7. RunAgentToLastContent 获取 Markdown 输出
8. 输出为空或非 Markdown 则返回错误
9. DocumentService.Create 保存文档

# 4. Agent 与提示词设计

## 4.1 Agent 组合
- 探索 Agent：api_explorer
- 校验 Agent：document_checker
- Markdown 校验：markdown_checker

## 4.2 提示词关键点
- 明确给出 YAML 线索作为背景信息
- 强调输出需按模块分表，并包含路径/功能/参数/说明
- 避免臆测，未找到的内容需说明

# 5. 代码改动范围

- 路由注册：`backend/internal/router/router.go`
- 前端触发：`frontend/src/pages/RepoDetail.tsx`、`frontend/src/services/api.ts`

# 6. 错误处理与日志
- 参数校验失败：返回明确错误
- Agent 执行失败：包装错误并记录 klog.V(6) 日志
- 输出为空：返回 ErrNoAgentOutput

# 7. 兼容性与安全
- 不修改现有任务状态机
- 不输出本地绝对路径到最终文档
- 文档生成仅基于源码事实与证据线索
