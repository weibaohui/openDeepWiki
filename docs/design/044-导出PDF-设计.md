# 1. 总体方案
在现有文档导出能力基础上，新增“导出 PDF”接口，将同一仓库下的全部文档内容按文档顺序拼接为单个 PDF，并在前端“查看文档”页的下载入口新增下拉选项，支持 ZIP 与 PDF 两种导出方式。

# 2. 接口设计

## 2.1 新增接口
- `GET /api/repositories/:id/export-pdf`：导出指定仓库的文档为 PDF

### 请求参数
- `id`：仓库 ID（路径参数）

### 响应
- `200 OK`：返回 `application/pdf` 二进制内容
- `400 Bad Request`：仓库 ID 非法
- `500 Internal Server Error`：导出失败

# 3. 后端实现设计

## 3.1 文档内容聚合
- 按仓库 ID 获取该仓库的文档列表，顺序与文档列表一致
- 为每个文档生成章节标题与内容拼接文本
- 空内容文档保留标题占位

## 3.2 PDF 生成
- 将拼接后的文本内容转换为 PDF
- 统一在服务端完成，返回二进制流
- 优先使用JetBrains Mono字体，注册失败时回退到Helvetica
- 优先使用中文字体，注册失败时回退到JetBrains Mono与Helvetica
- 支持标题、列表、引用、代码块、表格等基础Markdown样式
- 代码块检测到中文时使用中文字体渲染
- PDF 生成时为文档标题与Markdown标题创建目录书签
- 当文档标题与首个一级标题一致时，下调后续书签层级避免丢失子标题
- 保留文档中 a 标签链接为可点击跳转
- PDF 渲染逻辑抽离为独立服务，DocumentService 仅负责编排

# 4. 前端交互设计

## 4.1 下载入口
- 在“查看文档”页的下载按钮改为下拉菜单
- 提供“导出 ZIP”和“导出 PDF”两个选项
- 选择“导出 PDF”时触发下载并设置文件名

## 4.2 文件命名
- 文件名包含仓库名称与固定后缀，如：`{repoName}-docs.pdf`

# 5. 错误处理
- 后端错误返回明确错误信息
- 前端提示导出失败
