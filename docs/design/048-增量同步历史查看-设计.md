## 3. 后端设计

### 3.1 API 接口设计

#### 接口定义

- **路径**: `GET /api/repositories/:id/incremental-history`
- **描述**: 获取指定仓库的增量同步历史记录
- **参数**:
  - `id` (path): 仓库 ID
  - `limit` (query, optional): 返回记录数量限制，默认 50
- **响应**:
```json
{
  "code": 200,
  "message": "success",
  "data": [
    {
      "id": 1,
      "repository_id": 1,
      "base_commit": "abc123",
      "latest_commit": "def456",
      "added_dirs": 5,
      "updated_dirs": 3,
      "created_at": "2026-02-17T10:00:00Z"
    }
  ]
}
```

## 6. 数据流

1. 用户在仓库详情页点击"更多操作"按钮
2. 在抽屉菜单中点击"增量同步历史"按钮
3. 前端打开 `IncrementalHistoryDrawer` 组件
4. 组件挂载时调用 `repositoryApi.getIncrementalHistory(id)`
5. 后端 Handler 接收请求，解析参数
6. Service 调用 Repository 层查询数据库
7. Repository 执行 SQL 查询，返回记录列表
8. 后端返回 JSON 响应给前端
9. 前端渲染列表数据

## 7. 关键实现点

### 7.1 后端

1. **参数验证**: 确保 repositoryID 是有效的正整数
2. **权限检查**: 确保仓库存在且用户有权访问
3. **默认限制**: 如果未指定 limit，默认返回最近 50 条记录
4. **排序**: 按 created_at 倒序排列

### 7.2 前端

1. **状态管理**: 使用 useState 管理 loading、data 和 error 状态
2. **错误处理**: 捕获 API 错误并显示友好提示
3. **空状态**: 当没有历史记录时显示空状态组件
4. **响应式**: 确保 Drawer 宽度在不同屏幕尺寸下适配

### 7.3 性能优化

1. **延迟加载**: 仅在打开 Drawer 时才加载数据
2. **数据限制**: 通过 limit 参数控制返回数据量
3. **缓存**: 可考虑在前端缓存历史数据（可选）

## 8. 安全考虑

1. **SQL 注入**: 使用 GORM 的参数化查询，避免 SQL 注入
2. **越权访问**: 后端需验证仓库是否存在
3. **敏感信息**: Commit ID 不属于敏感信息，可直接展示
