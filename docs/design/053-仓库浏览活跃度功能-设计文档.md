# 053-仓库浏览活跃度功能-设计文档

## 技术方案设计

### 架构概述

采用事件驱动架构，通过事件总线解耦浏览事件的发生与活跃度计算逻辑。系统主要包含以下组件：

```
┌─────────────┐      Publish      ┌──────────────────┐
│   前端应用   │ ──────────────────>│  ActivityEventBus │
│ (未实现)    │                    └────────┬─────────┘
└─────────────┘                             │
                                            │ Subscribe
                                            ↓
                               ┌──────────────────────┐
                               │ ActivityEventSubscriber│
                               │  (更新活跃度数据)      │
                               └──────────────────────┘
                                            │
                                            ↓
                               ┌──────────────────────┐
                               │   RepoRepository      │
                               └──────────────────────┘

┌─────────────────────┐
│  ActivityScheduler  │
│ (定时检查并触发更新) │
└─────────────────────┘
```

### 关键组件说明

1. **ActivityEventBus**: 活跃度事件总线，负责发布和订阅浏览事件
2. **ActivityEventSubscriber**: 浏览事件订阅者，处理活跃度更新逻辑
3. **ActivityScheduler**: 调度器，定期检查并触发过期仓库的更新


### ActivityConfig配置结构

```go
type ActivityConfig struct {
    Enabled         bool          // 是否启用活跃度功能
    DefaultInterval time.Duration // 默认更新间隔 (7天)
    DecreaseUnit    time.Duration // 每个活跃点调减的时间 (1小时)
    CheckInterval   time.Duration // 定时检查间隔 (1小时)
    ResetHour       int           // 每日重置小时 (0-23)
}
```

### BrowseActivityEvent事件结构

```go
type BrowseActivityEvent struct {
    RepositoryID uint   // 仓库ID

}
```

## 活跃度计算逻辑

### 活跃度更新流程

1. 收到浏览事件
2. 检查是否需要重置今日活跃度（日期变更）
3. 增加活跃度点数
4. 计算新的更新间隔：`新间隔 = 默认间隔 - (活跃点数 × 单位调减时间)`
5. 限制最小间隔为1小时
6. 更新仓库的下一次更新时间

### 调度器检查流程

1. 定时触发（默认每小时）
2. 获取所有仓库
3. 检查每个仓库的 `NextUpdateTime` 是否已到达
4. 对于已到达的仓库，检查状态是否为 `completed`
5. 触发增量分析任务

### 活跃度重置规则

- 每日重置时间由 `ResetHour` 配置（默认凌晨0点）
- 当访问日期与 `LastActivityResetDate` 不同时重置
- 重置后 `TodayActivityCount` 归零，`LastActivityResetDate` 更新为当前日期

## 数据库变更

### GORM自动迁移

Repository模型新增的字段将通过GORM的自动迁移机制创建，无需手动执行SQL。

### 验证迁移

启动应用后，检查 `repositories` 表是否包含以下列：
- `next_update_time` (DATETIME, nullable)
- `today_activity_count` (INTEGER, default 0)
- `last_activity_reset_date` (DATETIME, nullable)

## 配置示例

### config.yaml

```yaml
activity:
  enabled: true
  default_interval: 168h  # 7天
  decrease_unit: 1h      # 每个活跃点调减1小时
  check_interval: 1h     # 每小时检查一次
  reset_hour: 0          # 凌晨0点重置
```

### 环境变量

当前实现不支持通过环境变量配置活跃度参数，需要修改 `config.go` 添加对应的环境变量读取逻辑。

## 已知限制

1. **前端集成未完成**：目前没有前端页面上报浏览事件
2. **增量分析触发未实现**：调度器中的增量分析触发逻辑仅为占位符
3. **多用户支持**：当前实现不区分不同用户的浏览行为，所有浏览行为统一计数
4. **配置持久化**：配置更新后不会持久化到文件，仅保存在内存中
