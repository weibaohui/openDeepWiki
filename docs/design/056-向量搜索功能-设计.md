## 一、需求概述

### 1.1 背景

当前 openDeepWiki 项目只能通过关键词搜索文档，无法实现基于语义的相似文章检索。为提升用户体验，需要引入向量搜索能力，为每篇文章生成向量嵌入，实现语义级别的相似性检索。

### 1.2 目标

实现完整的向量搜索功能，包括：
1. 向量数据模型和存储
2. 向量生成（支持 OpenAI Embeddings API）
3. 向量索引（HNSW 算法）
4. 语义搜索 API
5. 相似文章推荐 API
6. 异步向量生成任务

### 1.3 验收标准

1. 新文档保存后自动生成向量（异步）
2. 支持手动触发向量生成（单个/批量）
3. 支持语义搜索，返回相似度排序结果
4. 支持按文档 ID 查找相似文章
5. 向量生成失败不影响文档保存
6. 项目可正常编译和运行

## 二、技术设计

### 2.1 总体方案

采用分层架构，将向量功能独立成模块：
- **Model 层**：定义向量相关数据模型
- **Repository 层**：向量数据持久化
- **Domain 层**：向量领域模型和接口定义
- **Service 层**：向量生成、搜索业务逻辑
- **Handler 层**：HTTP API 处理
- **Pkg 层**：向量索引实现（HNSW）

### 2.2 技术选型

| 组件 | 选型 | 说明 |
|------|------|------|
| 向量存储 | SQLite BLOB | 利用现有数据库，无需额外依赖 |
| 向量索引 | HNSW（纯 Go） | 使用 `github.com/creachadair/go-hnsw` |
| 向量生成 | OpenAI Embeddings API | 支持 `text-embedding-3-small` (1536维) |
| HTTP 客户端 | 标准 `net/http` | 调用 OpenAI API |
| 异步任务 | Go goroutine + channel | 简单可靠的后台任务队列 |

### 2.3 数据模型设计

#### 2.3.1 数据库表结构

```sql
-- 文档向量表
CREATE TABLE document_vectors (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    document_id INTEGER NOT NULL,
    model_name TEXT NOT NULL,
    vector BLOB NOT NULL,           -- float32[] 二进制存储
    dimension INTEGER NOT NULL,
    generated_at DATETIME NOT NULL,
    metadata TEXT,
    FOREIGN KEY (document_id) REFERENCES documents(id) ON DELETE CASCADE
);

CREATE INDEX idx_doc_vectors_document_id ON document_vectors(document_id);
CREATE INDEX idx_doc_vectors_model ON document_vectors(model_name);

-- 向量生成任务表
CREATE TABLE vector_tasks (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    document_id INTEGER NOT NULL,
    status TEXT NOT NULL DEFAULT 'pending',
    error_message TEXT,
    created_at DATETIME NOT NULL,
    started_at DATETIME,
    completed_at DATETIME,
    FOREIGN KEY (document_id) REFERENCES documents(id) ON DELETE CASCADE
);

CREATE INDEX idx_vector_tasks_status ON vector_tasks(status);
CREATE INDEX idx_vector_tasks_document_id ON vector_tasks(document_id);
```

```go

type VectorIndex interface {
    // Add 添加向量到索引
    Add(id uint, vector []float32) error
```

### 2.9 异步任务处理

使用 goroutine + channel 实现简单的任务队列：

1. **任务入队**：文档保存成功后，将文档 ID 放入队列
2. **任务处理**：后台协程从队列取出文档 ID，生成向量
3. **错误处理**：生成失败时记录错误日志，不影响主流程
4. **状态跟踪**：通过 VectorTask 表跟踪任务状态

```go
// 任务处理流程
for docID := range s.taskQueue {
    go func(id uint) {
        // 1. 更新任务状态为 processing
        // 2. 获取文档内容
        // 3. 生成向量
        // 4. 保存向量
        // 5. 更新索引
        // 6. 更新任务状态为 completed/failed
    }(docID)
}
```

### 2.12 处理流程

#### 2.12.1 文档保存后自动生成向量

```text
文档保存成功
  ↓
发布 DocumentSaved 事件
  ↓
VectorEmbeddingService 监听事件
  ↓
创建 VectorTask（status: pending）
  ↓
将 docID 放入任务队列
  ↓
后台协程处理：
  1. 更新任务状态为 processing
  2. 获取文档内容（title + content）
  3. 调用 EmbeddingProvider 生成向量
  4. 保存向量到数据库
  5. 添加向量到索引
  6. 更新任务状态为 completed/failed
```

#### 2.12.2 语义搜索流程

```text
用户提交搜索请求
  ↓
Handler 验证参数
  ↓
VectorSearchService.Search()
  ↓
1. 调用 EmbeddingProvider 生成查询向量
  ↓
2. 调用 VectorIndex.Search() 获取相似文档 ID
  ↓
3. 从数据库获取文档详细信息
  ↓
4. 应用过滤条件（repository_id 等）
  ↓
5. 按相似度排序
  ↓
6. 返回结果
```

### 2.13 关键决策

| 决策点 | 选项 | 选择 | 理由 |
|--------|------|------|------|
| 向量存储 | SQLite BLOB / 外部库 | SQLite BLOB | 利用现有数据库，无额外依赖 |
| 索引算法 | HNSW / IVF / Flat | HNSW | 平衡准确性和性能 |
| 索引库 | creachadair/go-hnsw / 自研 | creachadair/go-hnsw | 成熟稳定，维护良好 |
| 任务队列 | Redis / channel | channel | 简单可靠，无需外部依赖 |
| 向量更新策略 | 替换 / 保留版本 | 替换 | 简化逻辑，节省存储 |

### 2.14 性能考虑

1. **索引构建**：首次启动时从数据库加载所有向量构建索引
2. **批量生成**：支持批量 API，减少网络开销
3. **索引持久化**：支持保存/加载索引，避免每次重启重建
4. **并发控制**：限制并发向量生成数量，避免资源耗尽

### 2.15 错误处理

1. **向量生成失败**：记录错误日志，任务状态标记为 failed
2. **搜索失败**：返回空结果，不中断服务
3. **索引损坏**：支持重新构建索引
4. **API 限流**：实现重试机制，避免临时失败
