## 一、需求概述

### 1.1 背景

当前 openDeepWiki 项目已经实现了代码仓库的智能解读和文档生成，并基于向量搜索实现了语义检索能力。用户希望能够基于已分析的代码仓库进行进一步的提问和对话，通过自然语言与代码仓库进行交互，获取更精准、更个性化的答案。

### 1.2 目标

实现完整的 AI 对话聊天问答功能，包括：
1. 基于单个仓库的智能问答对话
2. 支持多轮对话，记住上下文
3. 基于 RAG（检索增强生成）技术，利用向量搜索检索相关文档
4. 在回答中显示引用的文档来源，支持跳转
5. 对话会话管理（创建、删除、查询）
6. 对话历史记录
7. 前端聊天界面

### 1.3 功能边界

- **范围**：仅针对单个仓库进行对话，不支持跨仓库对话
- **对话模式**：支持多轮对话，系统会记住上下文
- **引用来源**：回答中必须标注引用的文档片段，用户可以点击跳转到原文档
- **数据来源**：基于已生成的文档内容，不直接访问代码文件

### 1.4 验收标准

1. 用户可以针对单个仓库创建对话会话
2. 用户可以在对话中提问，系统基于仓库文档给出答案
3. 系统支持多轮对话，能够记住上下文
4. 回答中显示引用的文档来源，用户可以点击跳转
5. 用户可以查看对话历史记录
6. 用户可以删除对话会话
7. 项目可正常编译和运行

## 二、技术设计

### 2.1 总体方案

采用 RAG（检索增强生成）架构，结合现有的向量搜索能力，实现智能问答：

- **Model 层**：定义对话会话和消息数据模型
- **Repository 层**：对话数据持久化
- **Service 层**：RAG 问答服务、对话管理服务
- **Handler 层**：HTTP API 处理
- **前端层**：聊天界面组件

### 2.2 技术选型

| 组件 | 选型 | 说明 |
|------|------|------|
| 对话存储 | SQLite | 利用现有数据库，无需额外依赖 |
| 向量检索 | 现有 VectorSearchService | 复用已有的向量搜索能力 |
| 问答生成 | LLM API（OpenAI 兼容） | 使用现有的 LLM 配置 |
| 上下文管理 | 滑动窗口 | 保留最近 N 轮对话 |
| 前端组件 | Ant Design Chat | 使用 Ant Design 的聊天组件 |

### 2.4 数据模型设计

#### 2.4.1 数据库表结构

```sql
-- 对话会话表
CREATE TABLE chats (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    repository_id INTEGER NOT NULL,
    title TEXT NOT NULL,
    created_at DATETIME NOT NULL,
    updated_at DATETIME NOT NULL,
    FOREIGN KEY (repository_id) REFERENCES repositories(id) ON DELETE CASCADE
);

CREATE INDEX idx_chats_repository_id ON chats(repository_id);

-- 对话消息表
CREATE TABLE chat_messages (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    chat_id INTEGER NOT NULL,
    role TEXT NOT NULL,           -- 'user' 或 'assistant'
    content TEXT NOT NULL,
    citations TEXT,                -- JSON 格式的引用信息
    created_at DATETIME NOT NULL,
    FOREIGN KEY (chat_id) REFERENCES chats(id) ON DELETE CASCADE
);

CREATE INDEX idx_chat_messages_chat_id ON chat_messages(chat_id);
CREATE INDEX idx_chat_messages_created_at ON chat_messages(created_at);
```

#### 2.4.2 Go 数据模型


### 2.5 RAG 服务设计

#### 2.5.1 RAG 流程

```
用户提问
  ↓
1. 构建上下文（获取最近 N 轮对话）
  ↓
2. 向量检索（基于问题检索相关文档）
  ↓
3. 构建 Prompt（问题 + 上下文 + 检索到的文档）
  ↓
4. 调用 LLM 生成答案
  ↓
5. 格式化答案（提取引用信息）
  ↓
6. 保存消息（用户消息 + 助手消息）
  ↓
7. 返回答案
```

### 2.13 关键决策

| 决策点 | 选项 | 选择 | 理由 |
|--------|------|------|------|
| 对话范围 | 单仓库 / 多仓库 | 单仓库 | 简化实现，符合用户需求 |
| 上下文管理 | 全量历史 / 滑动窗口 | 滑动窗口 | 控制成本，保留最近 N 轮 |
| 引用显示 | 仅标题 / 标题+片段 | 标题+片段 | 提供更多上下文信息 |
| 消息存储 | SQLite / 外部存储 | SQLite | 利用现有数据库，无额外依赖 |
| 前端组件 | 自研 / Ant Design | Ant Design | 成熟稳定，开发效率高 |

### 2.14 性能考虑

1. **向量检索**：复用现有的 VectorSearchService，已优化
2. **上下文管理**：限制上下文消息数量，避免 token 超限
3. **文档检索**：限制检索文档数量，平衡准确性和性能
4. **消息分页**：支持消息分页加载，避免一次性加载过多数据
5. **缓存策略**：缓存对话列表，减少数据库查询

### 2.15 错误处理

1. **向量检索失败**：返回空结果，记录错误日志
2. **LLM 调用失败**：返回错误信息，记录错误日志
3. **数据库操作失败**：返回错误信息，记录错误日志
4. **引用解析失败**：返回答案但不显示引用，记录错误日志

### 2.16 日志要求

使用 `klog.V(6)` 输出详细日志：

```go
klog.V(6).Infof("开始处理提问，chatID: %d, question: %s", chatID, question)
klog.V(6).Infof("检索到 %d 个相关文档", len(documents))
klog.V(6).Infof("生成答案成功，引用数量: %d", len(citations))
klog.Warningf("向量检索失败: %v", err)
klog.Warningf("LLM 调用失败: %v", err)
```

### 2.17 安全考虑

1. **输入验证**：验证用户输入，防止注入攻击
2. **权限控制**：确保用户只能访问自己仓库的对话
3. **敏感信息**：不在日志中记录敏感信息
4. **引用过滤**：过滤掉敏感文档的引用
