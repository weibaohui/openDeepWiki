# 3. 关键实现点

## 3.1 后端数据模型

**实现位置**：`backend/internal/model/user_request.go`

**关键代码**：
```go
type UserRequest struct {
    ID           uint      `json:"id" gorm:"primaryKey"`
    RepositoryID uint      `json:"repository_id" gorm:"index;not null"`
    Content      string    `json:"content" gorm:"type:text;not null;size:200"`
    Status       string    `json:"status" gorm:"size:50;not null;default:pending"`
    CreatedAt    time.Time `json:"created_at"`
    UpdatedAt    time.Time `json:"updated_at"`
    Repository   *Repository `json:"repository,omitempty" gorm:"foreignKey:RepositoryID"`
}
```

**设计要点**：
1. 使用 GORM 标签定义表结构和索引
2. Content 字段限制最大长度 200 字符
3. Status 字段设置默认值
4. 关联 Repository 模型，便于查询时获取仓库信息

## 3.2 Repository 层

**实现位置**：`backend/internal/repository/user_request_repo.go`

**设计要点**：
1. 使用接口定义，便于单元测试和替换实现
2. ListByRepository 方法支持分页和状态过滤
3. 使用 klog.V(6) 记录详细日志
4. 错误处理完善，包括数据库错误和记录不存在错误

## 3.3 Service 层

**实现位置**：`backend/internal/service/user_request_service.go`

**设计要点**：

## 3.4 Handler 层

# 4. 已知限制或待改进点

## 4.1 已知限制

1. **用户需求未与 Task 关联**
   - 说明：UserRequest 和 Task 保持独立
   - 影响：用户需求无法自动转换为文档生成任务
   - 应对：这是需求文档中明确声明的非目标

2. **没有频率限制**
   - 说明：没有对用户提交频率进行限制
   - 影响：可能被滥用，导致大量需求堆积
   - 改进：可以添加限流机制

3. **没有批量操作**
   - 说明：不支持批量删除用户需求
   - 影响：需要逐个删除，操作繁琐
   - 改进：可以添加批量删除功能

4. **没有搜索功能**
   - 说明：用户需求列表不支持关键词搜索
   - 影响：需求较多时查找困难
   - 改进：可以添加搜索功能

## 4.2 待改进点

1. **用户需求与 Task 的关联**
   - 可以考虑实现用户需求到 Task 的自动转换
   - 需求与 Task 的状态同步

2. **批量操作**
   - 可以添加批量删除、批量状态更新功能
   - 支持全选和批量操作

3. **用户反馈机制**
   - 可以添加用户对需求处理的反馈功能
   - 需求处理结果通知

4. **权限控制**
   - 可以添加权限控制，只有管理员可以删除需求
   - 用户只能查看自己的需求

# 6. 安全反思

## 6.1 安全检查

1. **输入验证**
   - ✅ 前端和后端都验证用户输入，确保内容非空且长度不超过 200 字符
   - ✅ 使用 Gin 的 binding 标签进行参数验证
   - ✅ Service 层进行二次验证

2. **SQL 注入防护**
   - ✅ 使用 GORM 参数化查询，避免直接拼接 SQL
   - ✅ 所有查询都使用参数绑定

3. **XSS 防护**
   - ✅ 前端使用 React 自动转义
   - ✅ 用户输入在显示时会被 React 自动转义

4. **越权访问防护**
   - ✅ 删除操作前验证需求存在性
   - ✅ 用户只能删除关联到其仓库的需求（通过 repository_id 过滤）

5. **日志记录**
   - ✅ 所有用户请求操作都记录到日志
   - ✅ 使用 klog.V(6) 输出详细日志
   - ✅ 错误信息记录到日志

## 6.2 风险评估

1. **无授权访问风险**
   - 风险：当前未实现用户认证，任何用户都可以提交和删除需求
   - 影响：用户可以删除其他用户的需求
   - 应对：建议后续添加用户认证和权限控制

2. **拒绝服务风险**
   - 风险：未实现频率限制，可能被恶意用户大量提交
   - 影响：数据库负载增加，影响正常用户使用
   - 应对：建议添加频率限制（如：同一仓库 1 分钟内只能提交 1 次需求）

3. **数据泄露风险**
   - 风险：需求内容可能包含敏感信息
   - 影响：敏感信息被记录到日志
   - 应对：建议对敏感信息进行脱敏处理
