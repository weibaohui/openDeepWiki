# 1. 背景（Why）

当前系统（030-用户内容需求反馈）已经实现了用户在浏览文档时提出需求的功能，但该实现仅将用户需求作为 Task 记录存储，没有独立的用户需求管理功能。为了更好地收集和管理用户提出的需求，需要：

1. 创建独立的 UserRequest 数据模型，而不是仅使用 Task 模型
2. 提供用户需求列表页面，方便查看和管理所有用户提出的需求
3. 在仓库详情页（/repo/:id）添加查看用户需求的入口链接
4. 完整的后端 UserRequest API 支持

# 2. 目标（What，必须可验证）

- [ ] 在文档查看页面的左侧文档列表末尾保留"没找到想要的"按钮（030 已实现）
- [ ] 点击按钮后弹出对话框，让用户填写一句话需求（030 已实现）

- [ ] 创建用户需求列表页面，显示所有用户提出的需求（新增）

- 不实现用户需求的批量导出功能
- 不实现用户需求的统计分析功能
- 不实现用户需求的优先级排序功能
- 不实现用户需求的自动执行和文档生成（由 Task 系统处理）
- 不实现用户需求的编辑功能

# 4. 使用场景 / 用户路径

## 场景 1：用户提交需求

```
用户访问文档查看页面 /repo/5/doc/10
                ↓
用户浏览左侧文档列表，发现没有想要查看的内容
                ↓
用户点击文档列表底部的"没找到想要的"按钮
                ↓
系统弹出对话框，提示"请用一句话描述您想要查看的内容"
                ↓
用户输入内容，例如："我想了解这个项目的认证机制是如何实现的"
                ↓
用户点击"提交"按钮
                ↓
系统将用户需求保存到 UserRequest 数据表
                ↓
系统显示"需求已提交"的成功提示
```

## 场景 2：管理员查看用户需求

```
管理员访问仓库详情页 /repo/5
                ↓
管理员点击"用户需求"链接
                ↓
系统跳转到用户需求列表页面 /repo/5/user-requests
                ↓
页面显示该仓库的所有用户需求
                ↓
管理员可以查看每个需求的详细信息
                ↓
管理员可以删除不需要的需求
```

# 6. 约束条件

## 6.1 技术约束

- 前端技术栈：React 19 + TypeScript + Ant Design 6
- 后端技术栈：Go 1.24+ + Gin + GORM
- 数据库：SQLite（默认）/ MySQL
- 使用 klog.V(6) 输出详细日志

## 6.2 架构约束

- 遵循项目分层架构（Handler -> Service -> Repository）
- 遵循错误处理规范
- UserRequest 与 Task 保持独立，不混用数据模型
- 复用现有的 Repository 模型和 RepositoryService

## 6.3 安全约束

- 对用户输入进行长度限制（最大 200 字符）
- 防止 SQL 注入（使用 GORM 参数化查询）
- 防止 XSS 攻击（前端转义用户输入）
- 记录所有用户请求到日志

## 6.4 性能约束

- API 响应时间不超过 1 秒（不包含 LLM 调用）
- 列表分页，每页最多显示 50 条记录
- 支持并发请求（至少 10 个/秒）

# 8. 接口与数据约定

## 8.1 UserRequest 数据模型

```go
type UserRequest struct {
    ID           uint      `json:"id" gorm:"primaryKey"`
    RepositoryID uint      `json:"repository_id" gorm:"index;not null"`
    Content      string    `json:"content" gorm:"type:text;not null;size:200"`
    Status       string    `json:"status" gorm:"size:50;not null;default:pending"`
    CreatedAt    time.Time `json:"created_at"`
    UpdatedAt    time.Time `json:"updated_at"`

    // 关联
    Repository   *Repository `json:"repository,omitempty" gorm:"foreignKey:RepositoryID"`
}
```

## 8.2 API 接口定义

### POST /api/repositories/:id/user-requests

创建用户需求

#### 请求
- **Path Parameters**:
  - `id`: 仓库 ID（integer）

- **Request Body**:
  ```json
  {
    "content": "我想了解这个项目的认证机制是如何实现的"
  }
  ```

#### 响应
**成功响应**（200 OK）:
```json
{
  "code": 0,
  "message": "需求已提交",
  "data": {
    "id": 1,
    "repository_id": 5,
    "content": "我想了解这个项目的认证机制是如何实现的",
    "status": "pending",
    "created_at": "2026-02-17T10:00:00Z",
    "updated_at": "2026-02-17T10:00:00Z"
  }
}
```

### GET /api/repositories/:id/user-requests

获取仓库的用户需求列表

#### 请求
- **Path Parameters**:
  - `id`: 仓库 ID（integer）

- **Query Parameters**:
  - `page`: 页码（默认 1）
  - `page_size`: 每页数量（默认 20，最大 50）
  - `status`: 状态过滤（可选）

#### 响应
**成功响应**（200 OK）:
```json
{
  "code": 0,
  "message": "success",
  "data": {
    "list": [
      {
        "id": 1,
        "repository_id": 5,
        "content": "我想了解这个项目的认证机制是如何实现的",
        "status": "pending",
        "created_at": "2026-02-17T10:00:00Z",
        "updated_at": "2026-02-17T10:00:00Z"
      }
    ],
    "total": 1,
    "page": 1,
    "page_size": 20
  }
}
```

### GET /api/user-requests/:id

获取用户需求详情

#### 请求
- **Path Parameters**:
  - `id`: 用户需求 ID（integer）

#### 响应
**成功响应**（200 OK）:
```json
{
  "code": 0,
  "message": "success",
  "data": {
    "id": 1,
    "repository_id": 5,
    "content": "我想了解这个项目的认证机制是如何实现的",
    "status": "pending",
    "created_at": "2026-02-17T10:00:00Z",
    "updated_at": "2026-02-17T10:00:00Z",
    "repository": {
      "id": 5,
      "name": "openDeepWiki",
      "url": "https://github.com/weibaohui/openDeepWiki"
    }
  }
}
```

### DELETE /api/user-requests/:id

删除用户需求

#### 请求
- **Path Parameters**:
  - `id`: 用户需求 ID（integer）

#### 响应
**成功响应**（200 OK）:
```json
{
  "code": 0,
  "message": "删除成功",
  "data": null
}
```

# 10. 风险与已知不确定点

## 已知风险
1. **数据一致性风险**
   - 说明：UserRequest 和 Task 可能需要关联
   - 影响：可能导致数据不一致
   - 应对：本版本暂不实现关联，后续根据实际需求考虑

2. **用户需求激增**
   - 说明：可能短时间内收到大量用户需求
   - 影响：列表页面性能下降
   - 应对：实现分页机制，限制每页数据量

## 不确定点
1. **用户需求的后处理流程**
   - 说明：用户需求提交后如何被处理和转化为文档
   - 应对：本版本仅实现需求收集和管理，不涉及后处理
