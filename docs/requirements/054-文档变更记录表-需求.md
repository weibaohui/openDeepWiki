# 1. 背景（Why）

当前系统中，所有AI生成的文档都没有统一的变更记录机制。随着文档的不断更新和迭代，用户难以了解文档的变更历史、修改人和修改时间等信息。为了更好地追踪文档的变更历史，需要：

1. 在所有文档的最后添加统一的"更新记录表"
2. 每次文档更新时，自动记录变更信息
3. 提供规范的表格格式，包含序号、修改人、主要修改内容、修改时间等字段
4. 更新文档智能体的提示词，确保生成的文档都包含此表格

# 2. 目标（What，必须可验证）

- [ ] 定义统一的文档变更记录表格式规范
- [ ] 更新 `document_generator.yaml` 智能体的提示词，要求生成的文档包含变更记录表
- [ ] 更新 `doc_rewriter.yaml` 智能体的提示词，要求重写的文档更新变更记录表
- [ ] 更新 `document_checker.yaml` 智能体的提示词，确保变更记录表的正确性
- [ ] 更新其他相关文档编写智能体的提示词
- [ ] 在需求设计文档和实现总结文档中应用此变更记录表格式

# 3. 非目标（Explicitly Out of Scope）

- 不实现文档版本控制系统
- 不实现文档对比功能
- 不实现文档回滚功能
- 不实现用户手动编辑变更记录表的界面
- 不实现变更记录表的数据库存储

# 4. 使用场景 / 用户路径

## 场景 1：AI生成新文档

```
用户请求AI生成新文档
                ↓
AI根据标题和代码仓库内容生成文档
                ↓
AI在文档最后添加"更新记录表"
                ↓
表格包含初始记录：序号1，修改人AI，修改时间，主要修改内容"初始版本"
                ↓
返回生成的完整文档
```

## 场景 2：AI重写/更新文档

```
用户请求AI更新文档内容
                ↓
AI读取原始文档，提取已有变更记录表
                ↓
AI根据重写指引更新文档内容
                ↓
AI在变更记录表中追加新的记录
                ↓
记录包含新序号、修改人、修改时间、主要修改内容
                ↓
返回更新后的完整文档
```

## 场景 3：文档校验

```
AI对文档进行校验
                ↓
检查文档是否包含变更记录表
                ↓
如果缺失，提示添加
                ↓
如果格式不正确，提示修正
                ↓
确保变更记录表符合规范
```

# 5. 功能需求清单（Checklist）

## 5.1 变更记录表格式规范

### 标准格式

```markdown
# 文件修改记录表

| 序号 | 修改人 | 修改时间 | 主要修改内容 |
| ---- | ------ | -------- | ------------ |
| 1 | AI | 2026-02-18 | 初始版本 |
```

### 字段说明

| 字段 | 类型 | 说明 | 示例 |
| ---- | ---- | ---- | ---- |
| 序号 | integer | 从1开始递增 | 1, 2, 3... |
| 修改人 | string | 修改者标识 | AI, 用户名等 |
| 修改时间 | date | YYYY-MM-DD格式 | 2026-02-18 |
| 主要修改内容 | text | 简要描述修改内容 | 初始版本、添加XXX功能、修复XXX问题 |

### 位置要求

- 变更记录表必须位于文档的最后
- 标题必须为 `# 文件修改记录表`
- 表格格式必须符合Markdown表格语法

# 6. 约束条件

## 6.1 技术约束

- 使用Markdown表格格式
- 日期格式必须为 YYYY-MM-DD
- 序号必须从1开始，连续递增
- 变更记录表位于文档开头

## 6.2 格式约束

- 表格标题固定为"文件修改记录表"
- 列标题固定为"序号|修改人|修改时间|主要修改内容"
- 列对齐方式：序号右对齐，修改人左对齐，修改时间左对齐，主要修改内容左对齐

## 6.3 内容约束

- 每次修改必须添加一条记录
- 记录内容要简明扼要，突出主要变化
- 修改人标识要清晰
- 修改时间要准确

# 7. 可修改 / 不可修改项

## ❌ 不可修改

- 变更记录表的基本格式（4列）
- 表格标题"文件修改记录表"
- 列标题名称
- 位置要求（文档开头）

## ✅ 可调整

- 修改时间的具体格式（可支持 YYYY-MM-DD HH:MM:SS）
- 主要修改内容的详细程度
- 是否添加额外字段（如影响范围、相关任务等）

## 8.1 变更记录表数据结构

## 8.2 智能体输入输出约定

### document_generator 输出要求

- 必须包含完整的变更记录表
- 初始记录序号为1
- 修改人为"AI"
- 修改时间为当前日期
- 主要修改内容为"初始版本"

### doc_rewriter 输入要求

- 原始文档内容（包含变更记录表）
- 重写指引

### doc_rewriter 输出要求

- 保留原始变更记录表
- 根据重写内容追加新的变更记录
- 序号正确递增
- 修改人、修改时间、修改内容准确

### document_checker 输入要求

- 待校验的文档内容

### document_checker 输出要求

- 修正后的文档内容
- 确保变更记录表符合规范

# 10. 风险与已知不确定点

## 已知风险

1. **序号冲突**
   - 说明：多人协作时可能出现序号冲突
   - 影响：记录顺序混乱
   - 应对：本版本仅处理AI场景，后续考虑协作场景

2. **内容重复**
   - 说明：可能出现重复的变更记录
   - 影响：记录冗余
   - 应对：智能体应检查并避免重复

## 不确定点

1. **修改时间格式**
   - 说明：是否需要支持更详细的时间格式
   - 应对：当前使用YYYY-MM-DD，后续可根据需要扩展

2. **修改人标识**
   - 说明：用户场景下的修改人标识方式
   - 应对：当前主要针对AI场景，用户场景待定
