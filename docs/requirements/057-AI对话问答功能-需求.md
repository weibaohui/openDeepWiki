## 1. 需求背景

当前系统主要基于 Agent 的文档生成模式，用户通过提交"需求"来触发 Agent 分析和文档生成。这种方式虽然能够生成高质量的文档，但缺乏实时交互性，用户无法在 Agent 思考和执行过程中进行干预或追加问题。

为了提升用户体验，需要实现一个类似 ChatGPT 的 AI 对话问答功能，让用户可以：
- 针对代码仓库进行实时问答
- 查看 AI 的思考过程和工具调用详情
- 在回答过程中进行干预（停止、追加输入）
- 保存完整对话历史供后续查看

---

## 2. 功能目标

### 2.1 核心目标

1. **实时对话交互**：用户能够针对代码仓库内容与 AI 进行实时对话
2. **透明化思考过程**：展示 AI 的思考过程和工具调用详情
3. **会话历史管理**：支持查看和恢复历史对话会话
4. **双向交互控制**：支持停止当前回答和追加输入

### 2.2 功能边界

**包含范围：**
- 基于代码仓库的问答对话
- 思考过程的可视化展示
- 工具调用结果的展示
- 会话的创建、保存、加载
- 实时停止和追加输入功能

**不包含范围：**
- 多轮对话的上下文压缩优化（第一期不做）
- 对话内容的分享功能（第二期）

## 3. 用户故事

### 3.1 故事一：发起对话

作为用户，我希望能够在仓库详情页发起 AI 对话，这样我可以针对当前仓库的代码提出问题并获得解答。

**验收标准：**
- 在仓库详情页提供"AI 对话"入口
- 点击后进入对话界面
- 可以输入问题并发送
- AI 基于当前仓库代码进行回答

### 3.2 故事二：查看思考过程

作为用户，我希望能够看到 AI 的思考过程和工具调用情况，这样我可以了解 AI 是如何得出结论的。

**验收标准：**
- 思考过程以可折叠的方式展示
- 每次工具调用都显示调用结果
- 工具调用结果包括：调用参数、返回结果、执行时间
- 思考过程按时间顺序展示

### 3.3 故事三：查看最终答案

作为用户，我希望看到 AI 最终给出的答案以清晰的格式展示，与思考过程区分开。

**验收标准：**
- 最终答案使用独立的回复块显示
- 支持 Markdown 格式渲染
- 代码块支持语法高亮
- 答案块可以复制

### 3.4 故事四：保存对话历史

作为用户，我希望我的对话能够被保存，这样我可以在后续查看之前的问答内容。

**验收标准：**
- 对话按 sessionID 保存到数据库
- 保存内容包括：用户提问、思考过程、工具调用、最终答案
- 保存时机：实时保存，AI每次处理都保存动作包括工具调用和调用结果
- 支持按仓库查看历史会话列表
- 点击历史会话可以再次查看完整对话内容

### 3.5 故事五：停止回答

作为用户，当 AI 的回答方向不对时，我希望能够立即停止当前回答，这样可以节省时间。

**验收标准：**
- 在 AI 回答过程中显示"停止"按钮
- 点击停止后，AI 立即停止生成
- 停止后保留已生成的内容
- 用户可以继续追加输入

### 3.6 故事六：追加输入

作为用户，我希望在 AI 回答过程中或回答完成后追加输入，让 AI 重新考虑后再次作答。

**验收标准：**
- 支持在回答过程中追加输入（会停止当前回答）
- 支持在回答完成后追加输入
- 追加输入后 AI 会结合上下文重新作答
- 追加输入的内容也保存到会话历史

## 4. 详细需求说明

### 4.1 数据模型设计

#### 4.1.1 会话表 (chat_sessions)

```go
type ChatSession struct {
    ID            uint           `gorm:"primaryKey"`
    SessionID     string         `gorm:"size:64;uniqueIndex;not null"` // 唯一会话标识
    RepositoryID  uint           `gorm:"index;not null"`               // 关联仓库ID
    Title         string         `gorm:"size:255"`                     // 会话标题（首条消息摘要）
    Status        string         `gorm:"size:20;default:'active'"`     // active, archived, deleted
    CreatedAt     time.Time
    UpdatedAt     time.Time
    Messages      []ChatMessage  `gorm:"foreignKey:SessionID;references:SessionID"`
}
```

#### 4.1.2 消息表 (chat_messages)

```go
type ChatMessage struct {
    ID            uint           `gorm:"primaryKey"`
    SessionID     string         `gorm:"size:64;index;not null"`       // 所属会话
    MessageID     string         `gorm:"size:64;uniqueIndex;not null"` // 唯一消息标识
    ParentID      *string        `gorm:"size:64;index"`                // 父消息ID（用于支持分支对话）
    Role          string         `gorm:"size:20;not null"`             // user, assistant, system, tool
    Content       string         `gorm:"type:text"`                    // 消息内容
    ContentType   string         `gorm:"size:20;default:'text'"`       // text, thinking, tool_call, tool_result
    ToolCalls     []ChatToolCall `gorm:"foreignKey:MessageID"`         // 关联的工具调用
    Model         string         `gorm:"size:50"`                      // 使用的模型
    TokenUsed     int            `gorm:"default:0"`                    // Token使用量
    Status        string         `gorm:"size:20;default:'completed'"`  // pending, streaming, completed, stopped, error
    CreatedAt     time.Time
    CompletedAt   *time.Time
}
```

#### 4.1.3 工具调用表 (chat_tool_calls)

```go
type ChatToolCall struct {
    ID            uint      `gorm:"primaryKey"`
    MessageID     string    `gorm:"size:64;index;not null"`       // 所属消息
    ToolCallID    string    `gorm:"size:64;uniqueIndex;not null"` // 工具调用标识
    ToolName      string    `gorm:"size:100;not null"`            // 工具名称
    Arguments     string    `gorm:"type:text"`                    // 调用参数（JSON）
    Result        string    `gorm:"type:text"`                    // 执行结果
    Status        string    `gorm:"size:20;default:'pending'"`    // pending, running, completed, error
    StartedAt     *time.Time
    CompletedAt   *time.Time
    DurationMs    int       `gorm:"default:0"`                    // 执行耗时（毫秒）
}
```

### 4.3 Agent 设计

#### 4.3.1 问答 Agent 定义

创建专门的问答 Agent：`backend/agents/chat_assistant.yaml`

```yaml
name: chat_assistant
description: 基于代码仓库回答用户问题的智能助手
instruction: |
  你是一个代码仓库分析助手，基于给定的代码仓库内容回答用户问题。

  ## 工作原则
  1. 先分析代码仓库结构，了解项目组织方式
  2. 针对用户问题，查找相关代码文件
  3. 基于实际代码内容给出准确回答
  4. 如果不确定，明确说明并建议查看的文件

  ## 工具使用规范
  - 使用 list_dir 了解目录结构
  - 使用 read_file 读取关键代码文件
  - 使用 search_files 查找特定模式

  - 使用 Markdown 格式
  - 代码引用使用代码块并标注语言
  - 复杂逻辑提供流程说明
  - 必要时提供文件路径链接

model: "default"
tools:
  - list_dir
  - read_file
  - search_files
  - git_log
  - git_show
maxIterations: 10

exit:
  condition: response_complete
  maxIterations: 10
```

### 4.5 实时交互设计

#### 4.5.1 停止功能

```
用户场景：
1. 用户发送问题
2. AI 开始思考和调用工具
3. 用户发现方向不对，点击"停止"按钮
4. 前端发送 stop 消息到 WebSocket
5. 后端收到 stop 信号，取消 Agent 执行
6. 前端显示"已停止"，保留已生成的内容
7. 用户可以追加输入或重新提问
```

#### 4.5.2 追加输入

```
用户场景：
1. AI 正在回答过程中
2. 用户输入追加内容，点击发送
3. 前端发送 stop 信号（如果有正在进行的回答）
4. 前端发送新的 message，包含 parent_id 指向当前对话节点
5. 后端收到新消息，基于上下文重新执行 Agent
6. 前端展示新的回答流
```

## 5. 技术实现要点

### 5.1 后端技术要点

1. **WebSocket 连接管理**
   - 使用 gorilla/websocket 或 nhooyr/websocket
   - 每个会话一个连接
   - 连接级别的心跳保活

2. **Agent 流式执行**
   - 复用 Eino ADK 的 Runner 机制
   - 拦截工具调用事件并发送到前端
   - 支持上下文取消（用于停止功能）

3. **消息持久化（实时保存）**
   - 用户消息：发送时立即保存
   - AI消息：创建消息记录（status=streaming），流式更新内容
   - 工具调用：每次调用开始时保存记录（status=running），完成后更新结果（status=completed）
   - 使用数据库事务保证消息和工具调用记录的完整性
   - 异步保存不阻塞流式输出，失败时记录日志但不中断对话
   - Token使用量统计完成后更新到消息记录

4. **并发控制**
   - 一个会话同时只能有一个正在进行的回答
   - 使用 sync.Mutex 或 channel 进行控制

### 5.2 前端技术要点

1. **WebSocket 连接管理**
   - 使用原生 WebSocket API
   - 实现断线重连逻辑
   - 页面离开时关闭连接

2. **流式渲染**
   - 使用 React 状态管理流式内容
   - 思考过程使用 Collapse 组件实现可折叠
   - 工具调用结果使用列表展示

3. **消息滚动**
   - 自动滚动到底部（用户未手动滚动时）
   - 加载历史消息时保持滚动位置

4. **状态管理**
   - 使用 React Context 或 Zustand 管理对话状态
   - 区分本地状态和服务器状态

## 7. 性能考虑

### 7.1 消息加载优化

- 历史消息分页加载，每页 20 条
- 使用虚拟列表渲染大量消息
- 工具调用结果懒加载（点击展开时加载详情）

### 7.2 流式输出优化

- 使用 requestAnimationFrame 节流渲染
- 工具调用结果缓存，避免重复查询
- WebSocket 消息压缩（可选）

---

## 8. 安全考虑

### 8.1 输入安全

- 对用户输入进行 XSS 过滤
- 限制单条消息长度（最大 4000 字符）
- 限制会话消息数量（最大 100 条）

### 8.2 权限控制

- 用户只能访问自己有权限的仓库会话
- 验证会话归属（session_id 与 repository_id 关联验证）

### 8.3 资源限制

- 限制单个会话的最大 Token 使用量
- 限制并发 WebSocket 连接数
- 限制 Agent 最大迭代次数

### 9.1 功能验收

- [ ] 用户可以在仓库详情页发起 AI 对话

- [ ] 用户无法访问其他用户的会话
- [ ] 输入过滤有效防止 XSS 攻击
- [ ] 资源限制有效防止滥用

| Thinking | 思考过程，AI 的推理和工具调用序列 |
| Stream | 流式输出，实时推送内容到前端 |
