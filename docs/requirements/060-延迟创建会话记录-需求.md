## 问题背景

当前实现中，用户点击"创建新对话"按钮时，系统会立即调用后端 API 在数据库中创建一条新的会话记录。这导致以下问题：

1. **空会话记录过多**：用户可能误点"创建新对话"或创建后未发送任何消息就离开，导致数据库中积累了大量空记录
2. **数据污染**：会话列表中显示了很多没有实际内容的会话，影响用户体验
3. **资源浪费**：无用的数据库记录占用存储空间

## 需求描述

### 功能目标

调整新会话的创建时机：
- **点击"创建新对话"**：仅在 UI 层面创建临时会话，不写入数据库
- **发送第一条消息时**：才真正调用 API 创建数据库记录并建立 WebSocket 连接

### 业务流程

```
用户点击"创建新对话"
    ↓
前端创建临时会话（本地状态，无 session_id）
    ↓
用户输入消息并点击发送
    ↓
检查是否为临时会话
    ↓
是 → 调用 API 创建真实会话 → 获取 session_id → 建立 WebSocket → 发送消息
否 → 直接使用现有 session_id → 发送消息
```

### 边界情况

1. **临时会话切换**：用户从临时会话切换到其他会话时，临时会话应被丢弃
2. **页面刷新**：临时会话不应持久化，刷新后消失
3. **多临时会话**：同时只允许存在一个临时会话
4. **发送失败**：如果创建会话 API 调用失败，应提示用户并重试

## 验收标准

- [ ] 点击"创建新对话"不调用后端 API
- [ ] 临时会话在 UI 上正常显示，可以输入消息
- [ ] 发送第一条消息时才创建数据库记录
- [ ] 发送成功后，会话列表更新显示新会话
- [ ] 切换到其他会话后，临时会话被正确清理
- [ ] 页面刷新后，临时会话消失
