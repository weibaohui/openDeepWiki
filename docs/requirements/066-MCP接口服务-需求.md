## 1. 需求背景

openDeepWiki 已经具备了：
- 代码仓库文档的自动生成能力
- 文档的存储和管理
- 基于 Web 的文档浏览和 AI 对话功能

但在 **VibeCoding** 场景下，开发者使用 AI 编程助手（如 Cursor、Claude Code、Windsurf）时，无法直接获取 openDeepWiki 中的文档知识。开发者需要：
- 在 IDE 中切换到浏览器查看文档
- 手动复制文档内容到 AI 对话中
- 无法让 AI 编程助手自动获取项目文档上下文

为了解决这个问题，需要提供一个标准化的接口，让 AI 编程助手能够直接查询 openDeepWiki 中的文档。

---

## 2. 功能目标

### 2.1 核心目标

1. **标准化接口**：提供 MCP (Model Context Protocol) 接口，兼容主流 AI 编程工具
2. **文档查询**：让 AI 能够发现、搜索和检索项目文档
3. **开发辅助**：AI 编程助手基于文档上下文辅助开发

### 2.2 功能边界

**包含范围：**
- MCP Server 实现（SSE 传输模式）
- 仓库列表和详情查询
- 文档搜索和全文读取
- 跨仓库查询支持

**不包含范围：**
- AI 问答能力（由调用方 AI 负责）
- 文档生成能力（通过 Web 界面操作）
- 认证功能（第一期假设内网部署，不提供认证）

## 3. 用户故事

### 3.1 故事一：配置 MCP 连接

作为开发者，我希望能够快速配置 AI 编程工具连接到 openDeepWiki，这样我的 AI 助手就能访问项目文档。

**验收标准：**
- 提供 SSE 端点 URL
- 支持 Cursor、Claude Code、Windsurf 等主流工具的配置格式
- 配置完成后 AI 能够成功连接

### 3.2 故事二：发现可用仓库

作为 AI 编程助手，我希望能够列出 openDeepWiki 中所有可用的仓库，这样我知道可以查询哪些项目的文档。

**验收标准：**
- 提供 `list_repositories` 工具
- 返回仓库名称、描述、状态、文档数量等信息
- 响应时间小于 1 秒

### 3.3 故事三：获取仓库文档列表

作为 AI 编程助手，我希望能够获取指定仓库的文档列表，这样我可以了解项目有哪些文档可以查询。

**验收标准：**
- 提供 `get_repository` 工具
- 支持通过 repo_id 或 repo_name 查询
- 返回文档标题、文件名、排序等信息

### 3.4 故事四：搜索文档

作为 AI 编程助手，我希望能够通过关键词搜索文档内容，这样我可以快速找到相关的文档片段。

**验收标准：**
- 提供 `search_documents` 工具
- 支持按关键词搜索
- 支持限定在特定仓库内搜索
- 返回匹配的文档片段和上下文

### 3.5 故事五：读取完整文档

作为 AI 编程助手，我希望能够读取完整的文档内容，这样我可以获取详细的上下文信息。

**验收标准：**
- 提供 `read_document` 工具
- 返回完整的 Markdown 格式文档内容
- 包含文档的元信息（所属仓库、分支、版本等）

## 4. 详细需求说明

### 4.1 MCP 工具定义

#### 4.1.1 list_repositories

列出所有可访问的仓库。

**输入参数**：无

**输出**：
```json
{
  "repositories": [
    {
      "id": 1,
      "name": "my-project",
      "description": "项目描述",
      "url": "https://github.com/org/my-project",
      "branch": "main",
      "status": "ready",
      "document_count": 12,
      "last_updated": "2026-03-15T10:00:00Z"
    }
  ]
}
```

#### 4.1.2 get_repository

获取仓库详情，包含文档列表。

**输入参数**：
```json
{
  "repo_id": 1,
  // 或
  "repo_name": "my-project"
}
```

**输出**：
```json
{
  "id": 1,
  "name": "my-project",
  "description": "项目描述",
  "url": "https://github.com/org/my-project",
  "branch": "main",
  "commit": "abc123",
  "status": "ready",
  "documents": [
    {
      "id": 101,
      "title": "API 设计",
      "filename": "api-design.md",
      "sort_order": 1
    }
  ]
}
```

#### 4.1.3 search_documents

搜索文档内容（关键词匹配）。

**输入参数**：
```json
{
  "query": "认证",
  "repo_id": 1  // 可选，不传则搜索所有仓库
}
```

**输出**：
```json
{
  "results": [
    {
      "doc_id": 101,
      "repo_id": 1,
      "repo_name": "my-project",
      "title": "API 设计",
      "filename": "api-design.md",
      "snippet": "...用户认证采用 JWT 方式..."
    }
  ],
  "total": 5
}
```

#### 4.1.4 read_document

读取文档完整内容（Markdown 格式）。

**输入参数**：
```json
{
  "doc_id": 101
}
```

**输出**：
```json
{
  "id": 101,
  "repo_id": 1,
  "repo_name": "my-project",
  "title": "API 设计",
  "filename": "api-design.md",
  "content": "# API 设计\n\n## 概述\n...",
  "branch": "main",
  "commit": "abc123",
  "version": 1
}
```

### 4.2 传输协议

采用 **SSE (Server-Sent Events)** 模式：
- 端点：`/mcp/sse`
- 协议：HTTP + SSE
- MCP 协议版本：2024-11-05

### 4.3 客户端配置格式

#### Cursor / Claude Code / Windsurf

```json
{
  "mcpServers": {
    "deepwiki": {
      "url": "http://your-server:8080/mcp/sse"
    }
  }
}
```

## 5. 非功能性需求

### 5.1 性能要求

| 指标 | 要求 |
|------|------|
| list_repositories 响应时间 | < 500ms |
| get_repository 响应时间 | < 500ms |
| search_documents 响应时间 | < 2s |
| read_document 响应时间 | < 1s（10KB 以内文档） |
| 并发连接数 | 支持 10+ 并发 |

### 5.2 可用性要求

- 内嵌到主服务，随主服务启动
- 服务异常时返回明确的错误信息
- 支持优雅关闭

### 5.3 安全要求（第一期）

- 假设内网部署，暂不提供认证
- 公网部署需要额外配置反向代理和认证
