# 068-MCP参数优化-需求

## 1. 问题背景

### 1.1 当前问题
当前 MCP 工具参数设计存在以下不符合 AI 使用习惯的问题：

1. **二选一参数困惑**：`get_repository` 的 `repo_id` 和 `repo_name` 二选一，AI 不知道优先用哪个
2. **无分页机制**：`list_repositories` 和 `search_documents` 返回全部结果，数据量大时 Token 消耗高
3. **过滤粒度不足**：`search_documents` 只能按仓库过滤，无法按文档类型、时间等过滤
4. **ID 获取路径长**：AI 需要多轮调用才能获取文档内容（list → get → read）

### 1.2 影响
- AI 调用效率低，需要多次交互
- 大仓库场景下响应过大，超出 AI 上下文限制
- AI 可能因参数困惑而调用失败

## 2. 需求目标

### 2.1 主要目标
优化 MCP 工具参数设计，使其更符合 AI 使用习惯，减少调用次数和 Token 消耗。

### 2.2 具体需求

#### R1: 消除二选一参数困惑
- `get_repository` 明确优先级：优先使用 `repo_id`，`repo_name` 作为备选
- 描述中明确说明 AI 应优先使用 ID

#### R2: 添加分页机制
- `list_repositories` 添加 `limit` 和 `offset` 参数
- `search_documents` 添加 `limit` 参数，默认返回 10 条，最大 50 条
- 返回结果包含 `total` 和 `has_more` 字段

#### R3: 增强过滤能力
- `search_documents` 添加 `doc_type` 参数，支持按文档类型过滤
- `list_repositories` 添加 `status` 参数，支持按状态过滤

#### R4: 减少调用路径
- `get_repository` 默认只返回文档摘要（id, title），不返回完整内容
- 添加 `include_content` 参数，控制是否返回文档完整内容

#### R5: 新增摘要工具
- 新增 `get_document_summary` 工具，返回文档前 500 字摘要
- 让 AI 快速判断文档相关性，减少无效完整读取

## 3. 非功能需求

### 3.1 性能
- 分页查询响应时间 < 100ms
- 摘要生成不额外查询数据库（从已有内容截取）

### 3.2 兼容性
- 保持现有 API 向后兼容
- 新增参数均为可选，不影响现有调用

## 4. 验收标准

| 编号 | 验收项 | 验证方式 |
|------|--------|----------|
| AC1 | AI 使用 `get_repository` 时优先使用 `repo_id` | 日志分析 |
| AC2 | 分页参数生效，返回结果包含 `has_more` | 单元测试 |
| AC3 | `search_documents` 默认返回 10 条 | 单元测试 |
| AC4 | 新增 `get_document_summary` 工具可用 | 手动测试 |
| AC5 | 现有 MCP 调用不受影响 | 回归测试 |

## 5. 优先级

P1 - 提升 AI 使用体验的关键优化

## 6. 预估工作量

- 开发：3 小时
- 测试：1 小时
- 文档：0.5 小时
