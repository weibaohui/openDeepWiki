# 053-仓库浏览活跃度功能

## 要解决的问题

当前仓库更新策略较为固定，所有仓库使用统一的更新间隔，无法根据用户的实际使用情况进行动态调整。这导致：

1. **资源浪费**：不活跃的仓库频繁更新，浪费计算资源和API调用配额
2. **响应延迟**：活跃的仓库更新间隔过长，用户无法及时获取最新内容
3. **缺乏用户行为反馈**：系统无法感知用户对哪些仓库更感兴趣

## 功能目标与边界

### 核心目标

1. **浏览活跃度追踪**：记录用户对各个仓库的浏览行为，统计活跃度点数
2. **动态更新间隔**：根据活跃度动态调整仓库的更新间隔，活跃的仓库更新更频繁
3. **活跃度管理**：提供配置界面，允许管理员调整活跃度计算参数
4. **定时检查机制**：定期检查仓库更新时间，到期自动触发更新任务

### 功能边界

- **范围**：
  - 追踪用户浏览仓库文档的行为
  - 动态调整已完成仓库的下一次更新时间
  - 提供活跃度配置API接口
  - 定时检查并触发过期仓库的更新

- **不在范围**：
  - 不追踪仓库的编辑、删除等其他操作
  - 不影响仓库的创建和初始分析流程
  - 不改变仓库分析任务本身的逻辑

### 核心设计思路

1. **事件驱动架构**：使用事件总线发布浏览活跃度事件，订阅者负责更新活跃度信息
2. **活跃度计算**：每次浏览增加一个活跃点，每个活跃点减少单位调减时间的更新间隔
3. **每日重置**：每日重置活跃度点数，避免长期积累导致更新间隔过短
4. **定时调度**：使用单独的调度器定期检查仓库更新时间，触发过期仓库的增量分析

1. **model/models.go**

2. **config/config.go**
   - 新增配置：ActivityConfig

3. **eventbus/** (新增)
   - activity_event.go: 活跃度事件总线定义

4. **subscriber/** (新增)
   - activity_event_subscriber.go: 活跃度事件订阅者

5. **service/** (新增)
   - activity_scheduler.go: 活跃度调度器

6. **handler/** (新增)
   - activity.go: 活跃度配置API处理器

7. **router/router.go**
   - 新增活跃度配置路由

8. **cmd/server/main.go**
   - 初始化活跃度事件总线、调度器和处理器

9. **service/repository.go**
   - Create方法中初始化活跃度字段
